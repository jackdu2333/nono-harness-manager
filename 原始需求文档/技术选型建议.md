# **NoNo Harness Manager 技术选型文档 v0.1**

## **1. 项目目标回顾**

NoNo Harness Manager 是一个 **macOS 桌面端本地 Agent Harness 管理器**。

它完整形态包含：

```text
Agents：Agent 客户端管理与一键启动
Skills：本地 Skill 扫描、分类、统计、元数据管理
MCP：MCP Server / Tools / Resources / Prompts 管理
Memory：本地记忆目录展示、分类、体检
Knowledge：Obsidian / VK / IMA 本地索引 / 项目知识库管理
Projects：项目维度绑定 Agent、Skill、MCP、Memory、Knowledge
Analytics：资源使用统计
Health Check：统一体检中心
```

但第一阶段只开发：

**Skills 管理面板 + 完整 Harness App 框架占位。**

# **2. 技术选型结论**

## **最终推荐技术栈**

```text
桌面框架：Tauri v2
前端框架：React + TypeScript
构建工具：Vite
UI 组件：shadcn/ui + Tailwind CSS
数据表格：TanStack Table
本地数据库：SQLite
Rust 数据访问：SQLx 或 rusqlite，推荐 SQLx
后端能力：Rust Tauri Commands
文件扫描：Rust walkdir / ignore
元数据解析：serde_yaml / serde_json
Markdown 预览：react-markdown
状态管理：Zustand
本地配置：SQLite + TOML / JSON 配置文件
日志：tracing / tauri-plugin-log
打包分发：Tauri Bundler
```

一句话结论：

**Tauri + React + TypeScript + SQLite + Rust 文件扫描，是当前最适合这个项目的组合。**

原因很简单：你要做的是本地文件扫描、本地数据库、Mac 桌面端、多目录权限、未来 CLI / MCP 扩展。Tauri 比 Electron 轻，Rust 后端比纯前端胡乱摸文件更靠谱，SQLite 足够稳。

# **3. 核心选型原则**

## **3.1 本地优先**

这个应用管理的是你本机的 Agent、Skill、MCP、记忆和知识库，所以默认应该：

```text
不联网
不上传
不依赖云端
不执行未知脚本
数据默认存在本机
```

## **3.2 安全优先**

NoNo Harness Manager 会访问大量本地路径，甚至未来可能启动 Agent、读取 MCP 配置、扫描记忆系统。

所以第一原则不是“功能强大”，而是：

**权限收口。**

Tauri v2 提供 capabilities 系统，用来细粒度限制前端窗口和 WebView 能访问的命令与插件能力；它的文件系统插件也强调通过 base directory 和路径 API 控制文件访问范围。这个设计非常符合你的本地资产管理器需求。 

## **3.3 阶段演进**

第一阶段只做 Skills 面板，不要为了未来 MCP Proxy、Memory AI 体检、跨客户端调用统计，一上来就把项目架构搞成“分布式航母战斗群”。

推荐：

```text
V0.1：单机桌面 App
V0.2：Agent 启动 + 使用统计
V0.3：MCP 管理
V0.4：Memory 目录展示 + 体检
V0.5：Knowledge / Projects
V1.0：CLI / Harness API / MCP Server
```

# **4. 桌面框架选型**

## **4.1 候选方案**

|**方案**|**优点**|**缺点**|**结论**|
|---|---|---|---|
|Tauri v2|轻量、安全边界好、Rust 后端、本地能力强|Rust 学习成本略高，部分插件生态不如 Electron|推荐|
|Electron|生态成熟、开发快、Node 文件系统方便|体积大、安全风险更重、资源占用高|不推荐作为首选|
|SwiftUI 原生|macOS 体验最好、系统能力最自然|开发成本高，和 Web/Rust/CLI/MCP 生态连接较麻烦|不适合第一阶段|
|Neutralino.js|极轻|生态和复杂本地能力不够强|不推荐|
|Flutter Desktop|跨平台 UI 稳|本地系统集成和 Web 技术复用不如 Tauri|暂不推荐|

## **4.2 推荐：Tauri v2**

Tauri 支持使用现有 Web 前端技术栈构建跨平台桌面应用，官方也明确支持任意前端框架；这让你可以用 React / TypeScript 快速做管理后台式 UI。 

Tauri 的优势：

```text
应用体积小
性能好
Rust 后端适合文件扫描
前端可用 React 快速开发
权限模型比 Electron 更收敛
后续可扩展 CLI / 本地 API / MCP
```

## **4.3 为什么不优先 Electron**

Electron 的确开发快，也有庞大生态。但 Electron 官方定义就是把 Chromium 和 Node.js 带到桌面端，能力大，体积也大。 

而你的应用要长期扫描本地目录、打开外部 App、读写元数据、管理 MCP 配置。Electron 不是不能做，而是需要非常严肃地处理安全边界。Electron 官方安全建议长期强调不要随便启用 Node 集成、要启用 context isolation、谨慎处理远程内容等。你这个项目第一阶段没必要背这个包袱。 

结论：

Electron 能做，但对 NoNo Harness Manager 来说偏重。除非你要极限追求前端生态开发速度，否则 Tauri 更合适。

## **4.4 为什么不优先 SwiftUI**

SwiftUI 是 Apple 官方推荐的声明式 UI 框架，适合构建 Apple 平台 App。 

但你的项目未来会涉及：

```text
Rust 文件扫描
SQLite
CLI
MCP Server
跨 Agent 调用
Web 技术 UI
可能跨平台
```

SwiftUI 做 Mac 原生体验很好，但不适合作为第一阶段最快落地方案。后期如果你想做一个更原生的“轻量菜单栏控制器”，可以单独考虑 SwiftUI 小组件。

# **5. 前端技术选型**

## **5.1 推荐组合**

```text
React + TypeScript + Vite + Tailwind CSS + shadcn/ui
```

## **5.2 React + TypeScript**

原因：

```text
适合做管理后台式界面
组件生态成熟
和 Tauri 集成成熟
你后续找 AI Agent 生成代码也最稳定
TypeScript 能减少数据结构混乱
```

## **5.3 Vite**

Tauri 官方文档中也有与 Vite 结合的路径，Vite 作为前端构建工具提供 HMR 和生产构建能力，开发体验比较适合桌面端前端。 

## **5.4 UI 组件：shadcn/ui + Tailwind**

推荐原因：

```text
界面干净
组件可复制进项目
容易被 AI 修改
适合管理控制台
不会被重型 UI 框架绑死
```

不建议第一阶段使用 Ant Design。

Ant Design 很强，但视觉上容易变成“公司后台管理系统”，你这个是 Mac 桌面端 Harness 控制台，不是进销存系统。别一打开就像在给仓库盘点螺丝。

## **5.5 数据表格：TanStack Table**

Skills、MCP、Memory、Knowledge 都会大量使用列表、筛选、分组、排序、分页。

TanStack Table 是 headless table 引擎，支持 rows、columns、sorting、filtering、grouping、pagination、selection 等能力，而且不接管你的 DOM 和样式。 

这点非常适合你：

```text
Skills 列表
MCP Tools 列表
Memory Items 列表
Knowledge Docs 列表
使用统计排行榜
健康检查结果列表
```

# **6. 后端与本地能力选型**

## **6.1 Tauri Commands + Rust**

推荐所有敏感本地能力都放在 Rust 后端：

```text
目录扫描
文件读取
文件写入
打开 Finder
启动 Agent 客户端
解析 skill.yaml
读写 SQLite
记录使用事件
健康检查
```

不要让前端直接拥有大范围文件系统权限。前端只通过 Tauri command 请求后端执行明确动作。

示例：

```text
scan_skill_source(source_id)
list_skills(filter)
get_skill_detail(skill_id)
save_skill_metadata(skill_id, yaml)
open_in_finder(path)
copy_agent_reference(skill_id)
launch_agent(agent_id)
record_usage_event(...)
```

## **6.2 文件扫描库**

推荐 Rust 侧使用：

```text
walkdir：递归扫描目录
ignore：支持 .gitignore 风格忽略规则
notify：后续做文件变化监听
```

第一阶段只做手动扫描。  
不要一开始就做实时监听，不然你会在 macOS 权限、性能、重复事件里快乐溺水。

## **6.3 元数据解析**

推荐：

```text
serde
serde_yaml
serde_json
toml
```

用于解析：

```text
skill.yaml
skill.json
MCP config
本地配置文件
```

# **7. 数据库选型**

## **7.1 推荐：SQLite**

SQLite 官方定义它是一个小型、快速、自包含、高可靠、功能完整的 SQL 数据库引擎；它也是 serverless、zero-configuration、transactional 的嵌入式数据库。 

这非常适合你的项目：

```text
单机应用
本地索引
资源元数据
使用统计
健康检查结果
无需部署数据库服务
备份简单
迁移简单
```

## **7.2 为什么不用 Postgres / MySQL**

因为这是本地桌面端应用。  
为了管理几个本地 Skill 去起一个数据库服务，像为了煮泡面修一座水电站。

## **7.3 Rust 数据访问库**

两个选择：

|**方案**|**优点**|**缺点**|**建议**|
|---|---|---|---|
|SQLx|async、支持 SQLite、查询可检查、适合长期项目|配置略复杂|推荐|
|rusqlite|简单直接、SQLite 专用|async 和工程扩展性弱一些|小项目可选|

SQLx 是 Rust 里的异步 SQL 工具包，支持 SQLite、PostgreSQL、MySQL 等，也支持编译期检查查询。 

推荐：

第一阶段用 SQLx + SQLite。

# **8. macOS 权限与系统集成选型**

## **8.1 文件权限**

你的 App 要扫描用户选择的目录，所以 macOS 权限是核心问题。

Apple 官方说明，沙盒 App 访问用户文件通常需要通过标准用户交互获得权限，并且如果要持久访问用户选择的文件系统资源，需要使用 security-scoped bookmarks。 

这意味着设计上必须遵守：

```text
用户主动选择目录
保存目录访问授权
重启后可恢复访问
路径失效时提示重新授权
不要默认扫描全盘
不要扫描敏感目录
```

## **8.2 打开 Finder**

通过 Tauri 后端 command 调用系统 opener 或 macOS open 命令：

```bash
open "/path/to/folder"
open -R "/path/to/file"
```

## **8.3 一键启动 Agent**

推荐第一阶段支持：

```bash
open -a "Codex"
open "/Applications/Codex.app"
```

后期再加：

```text
URL Scheme
CLI 启动
打开指定 workspace
启动时传参
```

# **14. 关键技术决策**

## **决策 1：桌面框架选择 Tauri v2**

理由：

```text
轻量
安全边界更清晰
Rust 后端适合本地扫描
前端可复用 React 生态
未来可扩展 CLI / MCP
```

## **决策 2：数据存 SQLite**

理由：

```text
本地单机最合适
无需服务
事务可靠
备份简单
查询足够强
适合索引和统计
```

## **决策 3：第一阶段不做实时文件监听**

理由：

```text
手动扫描更稳定
权限问题更少
调试成本更低
避免重复事件和性能问题
```

第二阶段再用 `notify` 做增量监听。

## **决策 4：第一阶段不执行脚本**

理由：

```text
安全边界复杂
执行日志复杂
权限提示复杂
容易误操作
```

先做资产管理，不做执行器。

## **决策 5：第一阶段不接 AI 自动分析**

理由：

```text
避免联网隐私问题
避免 prompt / key 管理
避免误改元数据
先把本地基础设施打稳
```

# **15. 安全设计要求**

## **15.1 路径安全**

必须做：

```text
只扫描用户添加的目录源
不默认扫描根目录 /
不默认扫描 ~ 全目录
忽略敏感目录
路径 normalize
防止路径穿越
写入前确认目标在授权目录内
```

敏感目录建议默认拒绝：

```text
~/.ssh
~/Library/Keychains
~/Library/Messages
~/Library/Mail
~/Library/Application Support/AddressBook
```

## **15.2 文件写入安全**

第一阶段只允许写：

```text
skill.yaml
skill.json
应用自己的 SQLite 数据库
应用自己的配置文件
```

写入前：

```text
备份原文件
校验 YAML
提示变更
记录操作事件
```

## **15.3 启动 Agent 安全**

启动 Agent 客户端只允许：

```text
用户明确配置的 app_path
用户明确配置的 launch_command
白名单命令 open
```

不要允许用户随便填复杂 shell：

```bash
rm -rf ~
curl xxx | sh
```

当然这类命令出现在配置里时，程序要像看见蟑螂一样立刻警告。

# **17. 第一阶段不选的技术**

## **不选 Electron**

不是不能做，而是对这个项目偏重。你要的是本地资产管理器，不是把半个 Chrome 塞进 Mac 里让风扇怀疑人生。

## **不选 SwiftUI 作为主栈**

原生体验好，但你当前更需要快速构建、多模块管理、AI 生成代码友好、后续 CLI / MCP 可扩展。

## **不选向量数据库**

第一阶段不做语义检索和相似度判断，不需要。

## **不选云数据库**

本地优先，不需要 Supabase、Firebase、MongoDB Atlas。  
为了本地 Skill 管理上云数据库，属于“拿火箭送外卖”。

## **不选自动脚本执行**

第一阶段风险太大，先只做管理。

# **19. 最终建议**

你的项目应该这样开局：

```text
大架构：NoNo Harness Manager
第一模块：Skills Panel
技术底座：Tauri v2 + React + SQLite
核心能力：扫描、索引、分类、统计、预览、元数据治理
安全原则：本地、只读、授权目录、后端封装
```

我不建议你再纠结 Electron / SwiftUI / Flutter。  
这类纠结通常不会让产品更好，只会让项目从“还没开始”变成“已经消耗三天但新建文件夹都没有”。

直接定：

**Tauri v2 + React + TypeScript + SQLite。**
