对，这次你说清楚了，**你要的是“日志扫描推断统计”，不是 MCP 观测，也不是主动上报，更不是 Harness UI 操作统计**。我前面把 MCP observed 和 agent_reported 也放进主计划里，确实扩得太宽了。那套适合未来，但不适合你当前这个需求。你现在要做的是：

**只基于各 Agent 客户端本地日志，推断 Skill / MCP 的使用痕迹，并做排行、矩阵、趋势。**

这个定位更窄，也更对。窄不是坏事，窄说明终于不用把整个宇宙塞进 SQLite 了，感谢理性短暂回归。

不过我建议保留字段里的 `event_source`，但第一阶段只允许一个值：

```text
log_inferred
```

其他 `mcp_observed / agent_reported` 可以先不实现，甚至不出现在 UI 里。留字段是为了未来不返工，当前产品口径只展示日志推断。现代可观测性里，日志通常是带 timestamp、severity 等上下文的事件记录，现代日志也常结构化以便机器解析；你这个方案本质就是把不同 Agent 客户端的日志转成统一事件表。

## **重新梳理后的核心原则**

这版计划应该定死三条边界：

**第一，只统计 Agent 客户端日志扫描结果。**  
Codex / Claude Code / WorkBuddy 日志里没有出现的，就不统计。Harness 不猜。

**第二，只做“日志推断调用”，不声称是真实调用。**  
UI 文案统一叫“可观测调用次数 / 日志推断 / 置信度”。不能写“真实调用次数”。日志解析天然存在漏报和误报，尤其 Skill 名称像 `doc`、`search`、`test` 这种普通词，简直是误报生产线。

**第三，Analytics 主排行排除 Harness UI 操作。**  
`view_detail / copy_path / open_folder / edit_description / archive / delete_index / scan` 这类一律不进 Agent 使用排行。它们可以留在审计日志或管理日志里，但不参与你这个 Analytics。

## **二、新增事件表**

建议新增表：

```sql
agent_resource_usage_events
```

字段：

```sql
id TEXT PRIMARY KEY,

agent_client TEXT NOT NULL,
agent_id TEXT,
agent_session_id TEXT,

resource_type TEXT NOT NULL,
-- skill / mcp_server / mcp_tool

resource_id TEXT,
resource_name TEXT NOT NULL,

-- skill_invoked

```

这里最重要的是 `event_hash UNIQUE`。日志扫描如果没有去重，今天扫一遍、明天扫一遍，调用次数就会原地通胀。人类经济已经够会通胀了，别让本地数据库也学坏。

## **三、采集方式**

第一阶段只做一种：

### **Log inferred**

为不同 Agent 客户端分别实现 adapter：

```text
CodexLogAdapter
ClaudeCodeLogAdapter
WorkBuddyLogAdapter
```

每个 adapter 做同一套动作：

```text
1. 定位日志目录
2. 扫描 session / transcript / tool call 文件
3. 根据 checkpoint 只处理新增内容
4. 匹配 Skill 名称、Skill 路径、MCP server 名称、MCP tool 名称
5. 生成 agent_resource_usage_events
6. 设置 confidence
7. 生成 event_hash 去重
```

事件日志分析里，case/session、activity、timestamp 这些关键属性非常重要，否则后面做趋势和矩阵会非常痛苦；相关过程挖掘研究也把 case-id、activity、timestamp 视为事件日志分析的核心线索。

## **四、匹配规则与置信度**

这是整个计划的命门。不要只做字符串 contains。  
字符串 contains 很勤奋，也很蠢。

### **High confidence**

满足任一：

```text
1. 日志中出现结构化 tool_call，且 tool 名称明确对应 MCP tool。
2. 日志中出现完整 Skill 路径。
3. 日志中出现完整 MCP server 名称或 tool call 记录。
4. 日志中出现明确字段，例如 skill_name / mcp_server / tool_name。
```

### **Medium confidence**

满足任一：

```text
1. 出现 Skill 名称，同时上下文附近出现 use skill / skill / prompt / tool / mcp 等关键词。
2. 出现 MCP server 名称，同时上下文附近出现 call / invoke / tool / server 等关键词。
3. 出现 Skill 名称 + Agent session 中存在相关执行上下文。
```

### **Low confidence**

满足任一：

```text
1. 只出现 Skill 名称。
2. 只出现很短的通用词，例如 doc、search、test、api。
3. 没有结构化调用证据。
```

主排行榜默认只显示：

```text
high + medium
```

低置信度不进入默认排行，只进入“低置信度待确认”区域。否则 `doc` 这种 Skill 会因为所有日志都出现“document/doc”而登顶，成为统计界的皇帝，新衣还挺透明。

## **五、扫描 checkpoint 与去重**

新增表：

```sql
agent_log_scan_checkpoints
```

字段建议：

```sql
id TEXT PRIMARY KEY,
agent_client TEXT NOT NULL,
source_file TEXT NOT NULL,
last_offset INTEGER,
last_mtime TEXT,
last_scanned_at TEXT,
created_at TEXT,
updated_at TEXT,

UNIQUE(agent_client, source_file)
```

扫描逻辑：

```text
1. 如果文件没扫过，从头扫描。
2. 如果文件已扫过，根据 last_offset 增量扫描。
3. 如果文件 mtime 变了但 offset 变小，认为日志轮转或重写，重新计算。
4. 每条识别事件生成 event_hash。
5. INSERT OR IGNORE 写入 agent_resource_usage_events。
6. 扫描完成后更新 checkpoint。
```

`event_hash` 建议基于：

```text
agent_client + source_file + source_offset + resource_type + resource_name + usage_kind + event_time
```

如果部分日志没有稳定 offset，就用：

```text
agent_client + source_file + event_time + normalized_evidence + resource_name
```

最近 7 天，按天：

```sql

```

### **8. trend_last_30_days**

最近 30 天，按天：


历史全量趋势。这里我建议**不要真的“全量按天”**，否则时间跨度一长，点位会很多。更合理：

```text

如果跨度 <= 2 年：按月
```

你计划里写“历史以来：全量聚合”，建议明确成：

```text
all_time_summary：总量
all_time_trend：根据跨度自动选择 day/month/year bucket
```

SQLite 的日期时间函数支持 `strftime()`，也支持把 ISO-8601 文本作为 time-value 来处理；你用 RFC3339 文本时间戳做 bucket 是可行的。

```sql

ON agent_resource_usage_events(resource_type, resource_id, event_time);

ON agent_resource_usage_events(event_source, event_time);

ON agent_resource_usage_events(confidence, event_time);
```

SQLite 查询规划器会利用索引减少扫描、排序和过滤成本，尤其是你这种后面要按时间范围、资源类型、客户端分组统计的查询。没有索引，日志一多，Analytics 就会开始用加载动画表达对人生的不满。
