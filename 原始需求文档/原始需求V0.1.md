**Mac 本地 Agent Harness 控制台**。

**Harness Manager 是一个 macOS 桌面端应用，用来统一管理本机 AI Agent 客户端、Skills、MCP、记忆系统、本地知识库，并提供启动、分类、统计、体检和调用入口。**

这比“Skill 管理器”高一层。Skill 面板只是第一阶段。  
好，人类终于不再把所有东西都塞进一个叫 `tools` 的文件夹里了，文明微弱进步。

## **产品名称**

暂定：

**Harness Manager**

## **产品类型**

macOS 桌面端 Agent Harness 管理器。

把你 Mac 上分散的 Agent 相关资产统一管理起来：

```text
Agent 客户端
Skills
```

# **三、核心概念重新定义**

## **1. Harness**

Harness 是总系统。

它不是某一个 Agent，也不是某一个 Skill，而是：

本机 AI Agent 运行环境的组织层、连接层和治理层。

## **2. Agent 客户端**

Agent 客户端是执行入口，比如：

- Codex
- Claude Code
- Cursor
- Qoder
- ChatGPT Mac 客户端
- WorkBuddy
- Gemini CLI
- 其他本地 Agent 客户端

## **3. Skill**

Skill 是能力单元，包括：

- Prompt
- Markdown Workflow
- Shell 脚本
- Python 脚本
- 项目模板
- 发布流程
- 部署流程
- 文档生成流程
- Obsidian 卡片生成流程

## **4. MCP**

MCP 是工具连接层。MCP 官方把它定义为连接 AI 应用与外部系统的开放标准，可连接数据源、工具和工作流；这正好对应你的 MCP 管理页定位。 

MCP 中的 Resources 用来向客户端暴露上下文数据，比如文件、数据库 schema 或应用数据，这也说明你的 MCP 页不能只看“工具”，还要看 resources、prompts、server 配置。 

## **5. Memory**

Memory 是上下文资产，不是普通文件夹。

它包括：

- MemOS
- rollout summaries
- historytalk
- closeout
- 项目记忆
- 用户偏好记忆
- Agent 运行摘要
- 长期记忆快照

## **6. Knowledge**

Knowledge 是资料资产，比如：

- Obsidian Vault
- VK 本地项目知识库
- IMA 云端知识库的本地索引
- 项目文档
- README
- Wiki
- 技术资料

# **四、新增需求逐条落地**

## **需求 1：Agent 客户端可以从 Harness 面板一键启动**

这个需求合理，而且应该作为 Agents 页面的一等功能。

macOS 本身有 Launch Services 机制，用来让应用打开其他应用、文档或 URL；Tauri 也有 opener 插件，可以打开文件、URL，或者在系统文件管理器里 reveal 文件。也就是说，“从 Harness 面板启动 Agent 客户端”技术上可行，但要做权限和路径配置。 

### **Agents 页面功能**

每个 Agent 客户端应支持：

- 登记名称
- 登记安装路径
- 登记配置目录
- 登记默认工作区
- 一键启动
- 打开配置目录
- 打开默认项目目录
- 显示是否已安装
- 显示最后启动时间
- 统计启动次数
- 绑定可用 Skills
- 绑定可用 MCP
- 绑定可用 Memory
- 绑定可用 Knowledge

### **Agent 数据字段**

```yaml
id: agent_codex
name: Codex
type: coding_agent
app_path: /Applications/Codex.app
launch_command: open -a Codex
config_path: ~/.codex
default_workspace: ~/Projects
supported_resources:
  - skills
  - mcp
  - memory
  - knowledge
status: active
last_launched_at:
launch_count: 0
```

### **一键启动策略**

建议分三种：

```text
方式一：通过 app_path 启动
方式二：通过 launch_command 启动
方式三：通过 URL scheme 启动，若客户端支持
```

MVP 先支持：

```bash
open -a "应用名"
open "/Applications/xxx.app"
```

第一版不要做复杂自动探测。先让用户手动添加客户端路径，避免“自动扫描 Applications 后把一堆奇怪应用识别成 Agent”，Mac 用户已经受够了各种智能误判。

## **需求 3：统计 Skills 和 MCP 的总使用次数，以及不同客户端的使用次数**

这是从“资产管理”升级到“资产运营”的关键功能。很高级，也很实用。

### **统计目标**

你需要知道：

```text
哪个 Skill 最常用？
哪个 Skill 从来没用过？
哪个 MCP 最常被 Codex 调用？
Claude Code 常用哪些 MCP？
哪些能力重复但没人用？
哪些能力最近变热？
```

这对后续清理、优化、归档非常重要。

### **Skills 统计指标**

每个 Skill 记录：

- 总使用次数
- 按 Agent 客户端分组的使用次数
- 最近使用时间
- 首次使用时间
- 最近 7 天使用次数
- 最近 30 天使用次数
- 使用来源
- 是否手动复制
- 是否被 Agent 引用
- 是否从详情页打开
- 是否执行过，后期再做

### **MCP 统计指标**

每个 MCP Server / Tool 记录：

- MCP Server 总调用次数
- 每个 Tool 调用次数
- 每个 Resource 读取次数
- 每个 Prompt 使用次数
- 按 Agent 客户端分组统计
- 最近调用时间
- 失败次数
- 平均响应时间，后期做
- 错误类型，后期做

### **使用统计表设计**

#### **resource_usage_events**

```sql
CREATE TABLE resource_usage_events (
  id TEXT PRIMARY KEY,
  resource_type TEXT NOT NULL, -- skill / mcp_server / mcp_tool / memory / knowledge / agent
  resource_id TEXT NOT NULL,
  agent_id TEXT,
  action TEXT NOT NULL, -- open / copy_path / copy_reference / launch / invoke / read / edit
  source TEXT, -- ui / cli / mcp / manual
  metadata TEXT,
  created_at TEXT NOT NULL
);
```

### **示例统计结果**

```text
Skills 使用统计

1. GitHub Release Skill
   总使用：128 次
   Codex：92 次
   Claude Code：31 次
   手动复制：5 次

2. Obsidian 知识卡片 Skill
   总使用：76 次
   ChatGPT：40 次
   Codex：21 次
   Claude Code：15 次

3. 阿里云部署 Skill
   总使用：42 次
   Codex：38 次
   手动复制：4 次
```

### **注意点**

第一阶段的“使用次数”不要幻想能自动捕捉所有 Agent 内部调用。除非 Agent 通过你的 CLI、MCP 或 Harness API 调用，否则你只能统计 Harness Manager 内发生的操作。

所以统计分三层：

|**层级**|**统计准确度**|**第一阶段能不能做**|
|---|---|---|
|UI 操作统计|高|能|
|CLI 调用统计|高|第二阶段|
|Agent 内部真实调用|取决于集成|后期|

MVP 先统计：

```text
打开详情
复制路径
复制 Agent 引用
打开 Finder
从 Harness 启动 Agent
```

后面有 CLI / MCP Server 后，再统计真实调用。

## **需求 5：增加记忆体检功能**

这个很有必要，而且会成为 Memory 页的核心价值。

“记忆体检”不要理解成医疗体检，当然我们还没疯到给 Markdown 做血常规。它是：

检查本地记忆系统是否健康、重复、过期、冲突、失联、不可读、过大、分类混乱。

### **记忆体检检查项**

#### **1. 路径检查**

- 记忆根目录是否存在
- 子目录是否存在
- 文件是否可读
- 文件是否为空
- 是否有断裂软链接
- 是否有权限错误

#### **2. 格式检查**

- Markdown 是否可读取
- JSON / JSONL 是否可解析
- YAML frontmatter 是否有效
- 文件编码是否正常
- 是否存在异常二进制文件

#### **3. 分类检查**

- 是否缺少分类
- 是否没有项目归属
- 是否无法识别记忆类型
- 是否存放在错误目录

#### **4. 重复检查**

- 文件名重复
- 内容高度相似
- 同一事实多处重复
- 同一项目多份摘要冲突

#### **5. 过期检查**

- 太久未更新
- 明显过期的项目记忆
- 已归档项目仍被 active Agent 引用
- 旧规则覆盖新规则

#### **6. 冲突检查**

比如：

```text
一条记忆说：VK 是默认知识库
另一条记忆说：IMA 是默认知识库
```

这种就要标记为冲突。

#### **7. 体积检查**

- 单个记忆文件过大
- JSONL 文件过大
- rollout summaries 数量过多
- closeout 未压缩
- 可归档文件过多

#### **8. 使用关系检查**

- 某个 Agent 配置引用了不存在的记忆目录
- 某个项目绑定了不存在的 memory path
- 某类记忆长期未被使用

### **体检输出形式**

```text
Memory Health Score：82 / 100

问题：
- 3 个路径失效
- 12 个文件缺少分类
- 5 个疑似重复记忆
- 2 个记忆事实冲突
- 1 个 JSONL 文件过大
- 4 个项目记忆超过 90 天未更新

建议：
- 归档旧 rollout summaries
- 合并重复的 ChatHub 项目记忆
- 修正 IMA / VK 默认知识库冲突
- 给未分类记忆添加 type 字段
```

### **体检结果表**

```sql
CREATE TABLE health_check_results (
  id TEXT PRIMARY KEY,
  target_type TEXT NOT NULL, -- memory / skill / mcp / knowledge
  target_id TEXT,
  check_type TEXT NOT NULL,
  severity TEXT NOT NULL, -- info / warning / error / critical
  title TEXT NOT NULL,
  description TEXT,
  suggestion TEXT,
  status TEXT DEFAULT 'open',
  created_at TEXT
);
```

# **NoNo Harness Manager 完整需求文档**

## **1. 项目背景**

用户正在构建自己的本地 AI Agent 工作流，电脑中存在多个 Agent 客户端、多套 Skills、多组 MCP 配置、本地记忆系统和本地知识库。

这些资源目前分散在不同目录和项目中，缺少统一管理入口，导致：

- 不知道有哪些 Agent 客户端。
- 不知道每个客户端能调用哪些资源。
- Skills 分散，难以复用。
- MCP 配置分散，缺少分类和状态管理。
- 记忆系统目录可见性差，缺少健康检查。
- 本地知识库和项目绑定关系不清晰。
- 缺少使用统计，不知道哪些资源真正有价值。

因此需要开发一个 macOS 桌面端 Harness 管理器。

---

## **2. 产品目标**

开发一个 macOS 桌面端应用 **NoNo Harness Manager**，用于统一管理本机 Agent Harness 生态。

完整目标：

```text
统一管理 Agent 客户端
统一管理 Skills
统一管理 MCP
统一管理记忆系统
统一管理本地知识库
统一管理项目绑定关系
提供分类聚合
提供使用统计
提供健康检查
提供一键启动和快速打开
```

---

## **3. 产品原则**

第一原则：

Harness Manager 管的是“本地 AI 工作环境”，不是某一种文件。

第二原则：

第一阶段只做 Skills 面板，但整体架构必须为 Agents、MCP、Memory、Knowledge 预留。

第三原则：

默认本地、默认只读、默认不联网、默认不执行脚本。

第四原则：

所有资源都要能分类、能搜索、能统计、能体检。

# **十一、最终架构判断**

你的最终架构可以这样定：

```text
NoNo Harness Manager
= 本地 Agent Harness 控制台

Agents
= 执行入口

Skills
= 能力单元

MCP
= 工具连接层

Memory
= 上下文资产

Knowledge
= 资料资产

Projects
= 组织单位

Analytics
= 使用价值判断

Health Check
= 可靠性治理
```

这版需求已经比较稳了。

最重要的是：**第一步只做 Skills 面板，但数据库、导航和概念模型必须按 Harness Manager 来设计。**

不要先做成一个孤立 Skill App，再未来“升级”为 Harness。那样会重构到怀疑人生。直接用 Harness 架构开局，Skills 只是第一个启用模块。
