# NoNo Harness Manager 最终落地需求

## 1. 需求转换结论

**NoNo Harness Manager 是一个 macOS 桌面端本地 Agent Harness 管理器。**

它不是单纯的 Skill 管理器，而是用来统一管理本机 AI Agent 运行环境的桌面控制台。完整形态会覆盖 Agent 客户端、Skills、MCP、Memory、Knowledge、Projects、Analytics、Health Check 等模块。

**第一阶段只落地 Skills 管理面板，同时搭好完整 Harness 应用框架和后续模块占位。**

这是当前原始需求中最关键的收敛点：不要第一阶段就做全功能平台，也不要把项目退化成单页 Skill 列表工具。

## 2. 产品定位

### 2.1 产品名称

暂定：

```text
NoNo Harness Manager
```

工程目录建议：

```text
nono-harness-manager
```

### 2.2 产品类型

macOS 桌面端本地 Agent Harness 管理器。

### 2.3 核心目标

统一管理本机分散的 AI Agent 资产：

```text
Agent 客户端
Skills
MCP Server / Tools / Resources / Prompts
Memory 记忆系统
Knowledge 知识库
Projects 项目绑定关系
Analytics 使用统计
Health Check 体检结果
```

### 2.4 核心原则

1. **本地优先**：默认不联网、不上传、不依赖云端。
2. **安全优先**：默认只读扫描，不执行脚本，不扫描未授权目录。
3. **阶段交付**：每个阶段都要能运行、能验收、能回滚。
4. **最小充分**：第一阶段只做 Skills MVP 和整体框架，不扩展隐藏功能。
5. **后端封装本地能力**：所有文件扫描、文件写入、Finder 打开、Agent 启动都通过 Rust Tauri command 完成。

## 7. Skill 识别规则

扫描 Skill 目录时，第一阶段按以下规则识别候选 Skill：

1. 文件夹内存在 `skill.yaml`，识别为标准 Skill。
2. 文件夹内存在 `skill.json`，识别为标准 Skill。
3. 文件夹内存在 `README.md`，识别为候选 Skill。
4. `.md` 文件识别为 Markdown / Prompt 型 Skill。
5. `.py`、`.sh`、`.js`、`.ts`、`.applescript` 识别为脚本型 Skill，但第一阶段绝不执行。
6. 文件名包含 `skill`、`prompt`、`workflow`、`agent`，识别为候选 Skill。

默认忽略：

```text
node_modules
.git
dist
build
__pycache__
.venv
.DS_Store
target
coverage
```

## 10. 安全约束

### 10.1 路径安全

必须满足：

```text
只扫描用户主动添加的目录源
不默认扫描根目录 /
不默认扫描 home 全目录
路径 normalize
防止路径穿越
写入前确认目标在授权目录内
忽略或拒绝敏感目录
```

敏感目录默认拒绝：

```text
~/.ssh
~/Library/Keychains
~/Library/Messages
~/Library/Mail
~/Library/Application Support/AddressBook
```

### 10.2 文件写入安全

第一阶段只允许写：

```text
skill.yaml
skill.json
应用自己的 SQLite 数据库
应用自己的配置文件
```

写入前必须：

```text
校验 YAML
备份原文件
确认目标在授权目录内
记录操作事件
```

### 10.3 脚本安全

第一阶段绝不执行扫描出的脚本型 Skill。

脚本文件只作为资产条目展示，不能作为命令运行。

### 10.4 Agent 启动安全

第一阶段不做 Agent 启动。

后续 Agent 启动只允许：

```text
用户明确配置的 app_path
用户明确配置的 launch_command
白名单 open 命令
```

不允许把复杂 shell 命令作为默认启动方式。

### 10.5 日志安全

日志不能记录：

```text
文件正文
记忆正文
敏感环境变量
密钥
Token
用户隐私内容
```

## 12. 开发纪律

后续每次开发必须遵守：

1. 开发前说明本次要改哪些文件。
2. 每次只完成一个清晰阶段。
3. 涉及数据库 schema 修改时，新增 migration，不直接改旧 migration。
4. 涉及文件写入时，说明写入路径和备份策略。
5. 涉及权限配置时，说明 Tauri capability 变更。
6. 涉及新依赖时，说明为什么需要它。
7. 不把临时 demo 代码混进正式结构。
8. 不把 mock 数据写死在业务逻辑里。
9. 不跳过 TypeScript 类型定义。
10. 不跳过 Rust 错误处理。
11. 不用 `unwrap()`、`expect()` 处理可恢复错误。
12. 不为了快速实现绕过 path guard。
13. 不把 UI、数据库、扫描逻辑混在一个文件里。
14. 每个阶段结束必须给出验收步骤。
15. 每次提交前必须确保项目能启动。

## 14. 待确认问题

这些问题不阻塞 Phase 0，但会影响 Phase 1 细节：

1. 默认预置哪些 Skill Sources：只空白引导，还是预置 `~/.codex/skills`、`~/.agents/skills`、`/Users/jackdu/memorydu` 等候选路径。
2. `copy_agent_reference` 的引用格式：是复制 Skill 路径、Skill 名称，还是生成某个 Agent 专用调用片段。
3. `skill.yaml` 备份命名规则：建议使用 `skill.yaml.bak.YYYYMMDD-HHMMSS`。
4. Skill 详情页是否允许直接编辑 Markdown 正文：建议第一阶段不做，只编辑元数据。
5. 第一阶段 Settings 基础页只放应用信息，还是允许配置数据库位置和默认扫描深度。

## 15. 后续最短执行路径

推荐下一步不要直接堆完整业务代码，而是按下面顺序执行：

```text
1. Phase 0：初始化 Tauri v2 + React + TypeScript + Vite 工程
2. 建立 AppShell、导航和占位页面
3. 接入 SQLite + SQLx migrations
4. 建立 Rust 分层目录和基础 command
5. 再进入 Phase 1：Skills MVP
```

**最短路径是先让桌面壳和数据库跑起来，再做 Skills 扫描。**

这样后续每个功能都能被真实应用承载，而不是散落成无法验收的 demo。
